### Shell 脚本错误处理与 trap 机制

上期答案: C
Jenkins声明式流水线: pipeline { agent any; stages { stage('Build') { steps {...} } } }
脚本式用 node {}，声明式用 pipeline {}

📚 [Shell] Shell 脚本错误处理与 trap 机制

>> 核心要点

1. 要点一：set -e 遇错即停 + set -o pipefail 管道错误也停——生产脚本第一道防线，避免带着错误继续执行导致数据损坏
2. 要点二：trap 'cleanup' EXIT——无论成功失败都执行清理，RTD 脚本中用于删临时文件、释放文件锁、回滚操作
3. 要点三：cmd || { echo "FAIL"; exit 1; } 精细错误处理——比 set -e 更灵活，适合需降级而非直接退出的场景

>> 一句话总结
Shell 脚本健壮性靠三道防线：set -e（全局）+ trap EXIT（兜底）+ &&/|| 显式检查（精细化）。

## 12:05 | 数据库

### MySQL 索引优化与 EXPLAIN 实战

上期答案: B (trap cleanup EXIT)

📚 [数据库] MySQL 索引优化与 EXPLAIN 实战

想象你去图书馆找一本《半导体工艺手册》。管理员告诉你书在"工业技术 → 无线电电子学 → 半导体技术 → 制造工艺"这个分类下。你拿着索书号 TN305，30 秒就找到了。如果没有这个分类系统，你得一排一排书架翻过去——这就是全表扫描 vs 索引查询的区别。

MySQL 的 B+Tree 索引就像一个多级分类目录。根节点是总目录，告诉你"去找 3 楼还是 5 楼"；内部节点是楼层指引，"去第 4 排还是第 7 排"；叶子节点才是真正放书的地方——而且叶子之间用链表串起来，范围查询（比如"TN305 到 TN310 所有书"）直接顺着链表走，不需要回根节点重新找。这就是 B+Tree 的精妙之处。

那怎么看一条 SQL 用没用上索引？EXPLAIN 就是你的 X 光机：

```sql
EXPLAIN SELECT * FROM wafer_lot WHERE lot_id = 'LOT20260525' AND status = 'ACTIVE';
```

输出里盯紧三个字段：
- type: 从好到坏是 const > ref > range > index > ALL。看到 ALL 就是全表扫描，红灯！
- key: 实际用的哪个索引。NULL 说明没走索引。
- rows: 预估扫描行数。越小越好，10 万+ 就该优化了。

RTD 场景里最经典的坑：你在 wafer_lot 表上给 lot_id 建了索引，但查询条件是 WHERE UPPER(lot_id) = 'LOT...'——函数一包，索引直接失效！MySQL 不认识 UPPER(lot_id) 对应的索引值，只能全表扫描。解决方案是把数据统一存成大写，或建函数索引（MySQL 8.0+）。

联合索引的"最左前缀"原则也常被忽略。建了 INDEX(lot_id, status, priority)，那么单独查 WHERE status='ACTIVE' 是不走索引的——就像你只有"省-市-区"的地址簿，只告诉你"区=南山区"却不说省市，没法定位。查询条件必须从索引最左列开始连续匹配才生效。

💡 记住一句话：索引是拿空间换时间，EXPLAIN 是拿一张表换清醒——慢查询不优化，等于给数据库穿小鞋。
