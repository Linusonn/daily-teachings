## 12:05 | 数据库

### MySQL 索引优化与 EXPLAIN 实战

📚 [数据库] MySQL 索引优化与 EXPLAIN 实战

想象你去图书馆找一本《半导体工艺手册》。管理员告诉你书在"工业技术 → 无线电电子学 → 半导体技术 → 制造工艺"这个分类下。你拿着索书号 TN305，30 秒就找到了。如果没有这个分类系统，你得一排一排书架翻过去——这就是全表扫描 vs 索引查询的区别。

MySQL 的 B+Tree 索引就像一个多级分类目录。根节点是总目录，内部节点是楼层指引，叶子节点才是真正放书的地方——而且叶子之间用链表串起来，范围查询直接顺着链表走。

EXPLAIN 就是你的 X 光机。盯紧三个字段：type（const>ref>range>index>ALL，ALL=全表扫描红灯）、key（NULL=没走索引）、rows（扫描行数越小越好）。

RTD 最经典的坑：WHERE UPPER(lot_id) = 'LOT...'，函数一包索引失效。联合索引最左前缀也常被忽略——INDEX(a,b,c)，单独查 WHERE b=1 不走索引。

💡 记住一句话：索引拿空间换时间，EXPLAIN 拿一张表换清醒。

## 13:02 | 中间件

### 消息队列 Kafka：半导体产线的神经中枢

上期答案: B (联合索引最左前缀) — INDEX(a,b,c) 查询必须从最左列 a 开始才能用上索引。WHERE a=1 AND c=2 跳过了 b 但仍可用 a 列过滤。



📚 [中间件] 消息队列 Kafka：半导体产线的"神经中枢"

先想一个场景：你管理的 Fab 厂里，光刻机每刻完一片晶圆就产生一条事件，蚀刻机每完成一个 recipe step 也产生一条，几百台设备每秒几千条消息——这些数据怎么汇聚、怎么路由到正确的消费方（RTD 调度系统、SPC 统计、MES 日志）？答案就是 Kafka。

Kafka 不是一个简单的"先进先出"队列，它像一个带分区的物流中心。你可以把 Topic 想象成不同的货品种类（比如 topic="tool-events" 收设备事件，topic="lot-tracking" 收批次轨迹）。每个 Topic 内部可以切多个 Partition——就像把一个大仓库分成了好几个小仓，每个小仓可以独立装货（写入）和出货（消费），所以并行吞吐量能做到百万条/秒。

关键角色就三个：Producer（设备上报事件的程序，比如 EAP 系统）、Broker（Kafka 服务器，存消息的）、Consumer（RTD 调度引擎、报警服务等）。Consumer 按 Group 组织——同一 Group 内的多个消费者分摊不同 Partition，实现负载均衡；不同 Group 之间各自独立消费同一份数据，互不影响。

在半导体 RTD 场景里，经典用法是这样的：EAP 把设备状态变更（IDLE→RUNNING、RUNNING→DOWN）写到 Kafka topic "eqp-state"，RTD 引擎的 Consumer Group 消费这些事件做实时调度决策，同时 BI 团队的另一个 Consumer Group 也消费同一份数据做 OEE 分析。互不干扰，各取所需。

Producer 发消息时可以指定 key（比如设备 ID），相同 key 的消息永远进同一个 Partition——这就保证了同一台设备的按序消费。对于报警联动场景（必须先收到"设备故障"再收到"故障恢复"），这个顺序保证至关重要。

💡 记住一句话：Kafka 不是队列，是分布式日志——消息不删，顺序写入，多消费者各自维护自己的"读到哪了"（offset）。

📚 [Linux] 文件描述符与 ulimit——为什么你的服务一到峰值就崩？

你在 Fab 厂里跑 RTD 系统，平时风平浪静。突然某个机台批量上报 Trace 数据，每秒 500 条——然后你的 Java 进程崩了，日志里写着 "Too many open files"。

这就好比一个餐厅只有 10 把椅子，平时够用。突然来了 50 个客人，后面的人只能在门外站着——在 Linux 里，"椅子"就是**文件描述符（File Descriptor, FD）**。

**什么是 FD？** Linux 哲学是「一切皆文件」——socket 连接是文件、管道是文件、甚至 /proc 下的进程信息也是文件。每打开一个，内核分配一个整数编号（FD）。默认上限：1024。

**RTD 场景真实案例**：假设你的 EAP 系统连接了 20 台机台的 SECS/GEM 端口（每台 2 个 socket），加上数据库连接池 20 个，再加上 HTTP 请求、日志文件、临时管道……轻松突破 1024。系统不是「撑不住」，是被内核硬生生卡死的。

**怎么查？**
```bash
ls /proc/$(pidof java)/fd | wc -l
cat /proc/sys/fs/file-max
ulimit -n
```

**怎么改？** 三种层面：
```bash
ulimit -n 65536
# 永久: /etc/security/limits.conf
* soft nofile 65536
* hard nofile 65536
# systemd: LimitNOFILE=65536
```

**两个坑**：1) ulimit -n 只在当前 shell 生效；2) soft limit 是实际生效值，生产环境 soft/hard 设成一样。

💡 记住一句话：Linux不会告诉你「快到上限了」——它只会在你撞墙的时候直接拒绝服务。

📚 [容器编排] Kubernetes 入门：从手工排班到自动化调度

想象你是 FAB 厂的排班主管，手下有 50 个工人（微服务），每天要手动安排谁去哪个机台、谁休息、谁顶班。更痛苦的是——有人突然"生病"（服务挂了），你得半夜爬起来顶岗。

Kubernetes（K8s）就是那个永不睡觉的排班 AI。你把工人名单交给它，它自动分配、自动替换、自动扩缩。

理解 K8s 只需要抓住三个核心：

1️⃣ Pod — 最小的调度单位
Pod 就是"一个工位"。通常一个 Pod 里跑一个容器（你的应用）。工人倒下？K8s 立马在原位拉一个新工人顶上——这叫"自愈"。

2️⃣ Deployment — 排班表
你告诉 K8s："我这个服务要 3 个副本。"Deployment 就确保永远有 3 个 Pod 在跑。想升级？一句 kubectl set image，K8s 会逐个替换（滚动更新），用户完全无感。

3️⃣ Service — 调度窗口
Pod 换来换去 IP 会变，Service 给一组 Pod 提供一个固定入口（ClusterIP），自动负载均衡。就像工厂里的"物料窗口"——不管里面谁在干，你只认这个窗口号。

实战三条命令感受一下：
```bash
# 跑一个 RTD 数据采集服务，3 副本
kubectl create deployment rtd-collector --image=rtd-collector:v1.2 --replicas=3

# 暴露服务端口
kubectl expose deployment rtd-collector --port=8080 --target-port=8080

# 滚动升级到新版本
kubectl set image deployment/rtd-collector rtd-collector=rtd-collector:v1.3
```

半导体 RTD 场景里，EAP、RMS、SPC 这些系统拆成微服务后，用 K8s 管理天然合适：机台报警时流量激增，HPA（水平自动伸缩）自动加 Pod；某个节点宕机，Pod 自动漂移到健康节点——这在人工运维时代根本不敢想。

💡 记住一句话：K8s 不替你写代码，但它替你的代码找地方住、帮它看病、给它排队——你只需告诉它"要几个、长啥样"。

Linux Shell | 管道、重定向与进程替换

先记住三个数字：0、1、2。是 Linux 给每个进程默认打开的三个文件描述符：0=stdin、1=stdout、2=stderr。

管道 | 是最常用的：ps aux | grep java。左边进程的 stdout 直接变成右边进程的 stdin，中间没有临时文件。

重定向：cmd > file 把 stdout 写进文件（覆盖），cmd >> file 是追加。cmd 2> err.log 把错误单独输出。最实用的是 cmd > out.log 2>&1——把 stderr 合并到 stdout 一起写。注意顺序不能反。

进程替换是 Bash 的隐藏大招。diff <(ls dir1) <(ls dir2) 把两个命令的输出当成虚拟文件传给 diff。RTD 场景里对比两台机台的 Trace 配置差异：diff <(ssh host1 cat /etc/trace/rules.conf) <(ssh host2 cat /etc/trace/rules.conf)——一行命令搞定。

tee 是管道里的分叉口：cmd | tee log.txt 把输出同时写到文件和管道下游。named pipe（mkfifo）适合两个独立脚本之间实时通信。

记住一句话：0 进 1 出 2 报错，管道串联，重定向分流。

📚 数据库 | MySQL 慢查询分析与索引优化实战

想象你在一个巨大的图书馆找一本书。没有索引，你得从第一排书架走到最后一排，一本一本看——这就是全表扫描。有索引就像有张卡片目录，直接告诉你"3楼 B区 第5排"——几十亿条记录的 Trace 数据库里，有索引和没索引，查询速度能差几万倍。

RTD 半导体场景里，每天产线产生的 Trace 数据动辄上亿条。你写了一个查询"某机台过去一周的 Temperature 曲线"，如果没建索引，一条 SQL 可能跑半小时，MES 界面直接卡白屏。

第一步：找到慢查询

MySQL 有个开关叫 `slow_query_log`：

```sql
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;   -- 超过1秒就算慢
SET GLOBAL log_queries_not_using_indexes = ON;
```

慢查询日志会记录执行时间超阈值的 SQL。生产环境别全开，用 `long_query_time = 0.5` 抓典型分析就行。

第二步：用 EXPLAIN 看执行计划

这是最重要的诊断命令：

```sql
EXPLAIN SELECT * FROM trace_data 
WHERE equipment_id = 'E-01' AND param = 'Temp'
  AND ts > '2026-05-20';
```

重点看三个关键字段：
- **type**: 从 ALL（全表扫描，最差）→ index → range → ref → const（最优）。出现 ALL 赶紧加索引
- **rows**: 预估扫描行数。1000 万行？那基本等于页面白屏
- **key**: 实际用到的索引名。NULL 表示没索引可用
- **Extra**: `Using filesort` 或 `Using temporary` = 性能杀手，说明 MySQL 被迫自己做了排序或临时表

EXPLAIN 还有个兄弟叫 `EXPLAIN ANALYZE`（MySQL 8.0.18+），不光显示计划，还告诉你每步实际花了多少毫秒，定位瓶颈一步到位。

第三步：建索引，但要聪明地建

```sql
ALTER TABLE trace_data ADD INDEX idx_eqp_param_ts (equipment_id, param, ts);
```

联合索引有「最左前缀」原则：索引 `(equipment_id, param, ts)` 对 `WHERE equipment_id = ?` 有效，对 `WHERE equipment_id = ? AND param = ?` 有效，但对 `WHERE param = ?`（跳过了第一列）无效。就像你不能跳过一楼直接走二楼的楼梯——开发商没这么建。

常见翻车现场：
- `WHERE name LIKE '%abc'` — 前导模糊，索引直接失效。变通：用全文索引，或把字符串反转存储后前缀匹配
- `WHERE status + 1 = 2` — 列上做运算，索引失效。改成 `WHERE status = 1`
- `WHERE DATE(create_time) = '2026-05-25'` — 函数包裹列，索引失效。改成范围查询 `WHERE create_time >= '2026-05-25' AND create_time < '2026-05-26'`

RTD 实战建议：Trace 表按 `equipment_id + param_name + timestamp` 建联合索引；Lot 流转表按 `lot_id + operation_seq` 建；设备状态变更表按 `equipment_id + event_time` 建。索引不是越多越好——每个索引都会拖慢 INSERT/UPDATE，只在查询真正需要的列上建。

💡 记住一句话：索引是数据库的地图——没有地图你只能徒步走遍全城，有了它打车直达。但地图太多也会占背包（存储）和拖慢脚步（写入性能）。
