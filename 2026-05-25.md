### MySQL 索引优化与 EXPLAIN 实战

上期答案: B (trap cleanup EXIT) — trap 'cleanup' EXIT 注册退出钩子，脚本正常结束或被信号终止时都会执行清理，比手动在末尾写 cleanup 更可靠。

📚 [数据库] MySQL 索引优化与 EXPLAIN 实战

想象你去图书馆找一本《半导体工艺手册》。管理员告诉你书在"工业技术 → 无线电电子学 → 半导体技术 → 制造工艺"这个分类下。你拿着索书号 TN305，30 秒就找到了。如果没有这个分类系统，你得一排一排书架翻过去——这就是全表扫描 vs 索引查询的区别。

MySQL 的 B+Tree 索引就像一个多级分类目录。根节点是总目录，内部节点是楼层指引，叶子节点才是真正放书的地方——而且叶子之间用链表串起来，范围查询直接顺着链表走。

EXPLAIN 就是你的 X 光机。盯紧三个字段：type（const>ref>range>index>ALL，ALL=全表扫描红灯）、key（NULL=没走索引）、rows（扫描行数越小越好）。

RTD 最经典的坑：WHERE UPPER(lot_id) = 'LOT...'，函数一包索引失效。联合索引最左前缀也常被忽略——INDEX(a,b,c)，单独查 WHERE b=1 不走索引。

💡 记住一句话：索引拿空间换时间，EXPLAIN 拿一张表换清醒。

## 13:02 | 中间件

### 消息队列 Kafka：半导体产线的神经中枢

上期答案: B (trap cleanup EXIT 是正确写法，双引号内是信号列表)

📚 [中间件] 消息队列 Kafka：半导体产线的"神经中枢"

先想一个场景：你管理的 Fab 厂里，光刻机每刻完一片晶圆就产生一条事件，蚀刻机每完成一个 recipe step 也产生一条，几百台设备每秒几千条消息——这些数据怎么汇聚、怎么路由到正确的消费方（RTD 调度系统、SPC 统计、MES 日志）？答案就是 Kafka。

Kafka 不是一个简单的"先进先出"队列，它像一个带分区的物流中心。你可以把 Topic 想象成不同的货品种类（比如 topic="tool-events" 收设备事件，topic="lot-tracking" 收批次轨迹）。每个 Topic 内部可以切多个 Partition——就像把一个大仓库分成了好几个小仓，每个小仓可以独立装货（写入）和出货（消费），所以并行吞吐量能做到百万条/秒。

关键角色就三个：Producer（设备上报事件的程序，比如 EAP 系统）、Broker（Kafka 服务器，存消息的）、Consumer（RTD 调度引擎、报警服务等）。Consumer 按 Group 组织——同一 Group 内的多个消费者分摊不同 Partition，实现负载均衡；不同 Group 之间各自独立消费同一份数据，互不影响。

在半导体 RTD 场景里，经典用法是这样的：EAP 把设备状态变更（IDLE→RUNNING、RUNNING→DOWN）写到 Kafka topic "eqp-state"，RTD 引擎的 Consumer Group 消费这些事件做实时调度决策，同时 BI 团队的另一个 Consumer Group 也消费同一份数据做 OEE 分析。互不干扰，各取所需。

Producer 发消息时可以指定 key（比如设备 ID），相同 key 的消息永远进同一个 Partition——这就保证了同一台设备的按序消费。对于报警联动场景（必须先收到"设备故障"再收到"故障恢复"），这个顺序保证至关重要。

💡 记住一句话：Kafka 不是队列，是分布式日志——消息不删，顺序写入，多消费者各自维护自己的"读到哪了"（offset）。
