📚 [Groovy] Groovy 闭包与 DSL 实战：RTD 工作流里的瑞士军刀

上期答案：C（联合索引最左前缀匹配原则——WHERE A=1 AND B=2 可以用到联合索引 (A,B,C)，因为 A 在最左边。MySQL 从最左列开始依次匹配，跳过最左列的话整个索引就废了，比如 WHERE B=2 AND C=3 就命中不了。）

---

不知道你有没有过这种体验：在产线上，晶圆在一片一片地跑，每一片该去哪台设备、跑什么配方、出什么参数，这些规则变起来比代码改版还频繁。你要是每次都用 Java 改 → 编译 → 打包 → 重启，产线早停了。

这就是 Groovy 在半导体 RTD 系统里的价值：**让业务规则可热加载**。

**闭包，就是一块你随时能打开的代码**

你可以把它想象成一把折叠伞。收起来递给同事：喏，这个工序逻辑你拿着。他到车间要用的时候，撑开就行。

```groovy
// 定义一个蚀刻工序模板
def etchStep = { wafer, params ->
    wafer.status = "ETCHED"
    log("${wafer.id} 蚀刻完成, 耗时 ${params.time}s")
}

// 用到时才执行
etchStep(wafer01, [time: 120, temp: 200])
```

闭包不只是「能传参的函数」，它有三个特殊属性：owner（定义它的对象）、delegate（方法查找目标对象）、resolveStrategy（找方法的优先级）。delegate 是 DSL 的灵魂——它让闭包里的方法调用「重定向」到另一个对象身上。

**DSL：用 Groovy 写「看起来像配置文件的东西」**

比如在 RTD 系统中，设备工程师想定义一批报警规则：

```groovy
alarmRules {
    forDevice("E-01") {
        when("温度 > 200") {
            severity "CRITICAL"
            notify "shift-leader@factory.com"
            action "HOLD_EQUIPMENT"
        }
        when("压力 < 0.1") {
            severity "WARNING"
            autoRecoverAfter 5.minutes  // Groovy 语法糖
        }
    }
}
```

这看起来是人写的配置，但它其实是合法的 Groovy 代码，背后靠的就是把闭包的 delegate 指向一个 DSL 上下文对象：

```groovy
def alarmRules(Closure block) {
    block.delegate = new AlarmDSLContext()
    block.resolveStrategy = Closure.DELEGATE_FIRST
    block.call()
}
```

当解析到 `when(...) { ... }` 时，Groovy 在闭包自己的类里找不到 `when` 方法 → 顺着 delegate 找 → 在 AlarmDSLContext 里找到并执行。

**在 RTD 场景里到底怎么用？**

调度引擎（哪台设备空闲了派哪个批次），MES 工序流转规则，设备 PM 保养触发条件，SPC 统计判定规则——这些都应该写成 .groovy 文件放在固定目录。系统通过 GroovyShell 或 GroovyScriptEngine 热加载，改动后无需重启：

```
运维改 .groovy → 文件时间戳变了 → 引擎自动重载 → 新规则生效
```

这就是半导体柔性制造的核心基础设施之一。

💡 记住一句话：**闭包是 Groovy 的肌肉，delegate 是骨架，DSL 是穿上西装后的样子——看着像配置，跑起来是代码。**

[网络] TCP KeepAlive 与心跳：半导体设备通信的「呼吸检测」

在半导体 Fab 里，RTD 系统通过 TCP 长连接跟几百台设备通信——EAP 发指令、收 SECS 消息、采集 Trace 数据。问题来了：网络断了你得立刻知道。不然设备挂了你还以为连着，Lot 卡在机台里没人管，这就是生产事故。

TCP 自带 KeepAlive 机制。就像你打电话时沉默太久，对方问「还在吗？」——TCP 空闲后发探测包，没回应就重试，再没回应就判死亡。

三个关键参数（Linux）：
net.ipv4.tcp_keepalive_time = 7200   # 空闲多久开始探测，默认2小时
net.ipv4.tcp_keepalive_intvl = 75    # 探测间隔（秒）
net.ipv4.tcp_keepalive_probes = 9    # 探测几次就放弃

默认要 7200+75*9=7875 秒（>2小时）才发现断连——在产线上完全不可接受！

所以 RTD 系统必须叠加应用层心跳：客户端每 10 秒发 PING，服务端回复 PONG。3 次没回（30秒）就触发重连。Netty 实战一行搞定：

ch.pipeline().addLast(new IdleStateHandler(0, 0, 10, TimeUnit.SECONDS));
// 10 秒没收到数据 -> userEventTriggered 发心跳帧

同时开启 Socket 级 KeepAlive 兜底：
bootstrap.option(ChannelOption.SO_KEEPALIVE, true);

TCP KeepAlive 是兜底安全网，应用层心跳才是生命线。在 RTD 场景，宁可误判重连（多几条日志），也不能漏判（卡 Lot 就是生产事故）。

记住一句话：TCP KeepAlive 默认 2 小时才发现断连，RTD 必须叠加应用层心跳才能守住产线。

## 11:03 | Linux 性能

### iostat + iotop：磁盘 I/O 的「心电图仪」— RTD Trace 落盘性能诊断

上期答案: B

tcp_keepalive_time 的默认值是 7200 秒（2 小时），含义是 TCP 连接在空闲多久后开始发送第一个 KeepAlive 探测包。这个参数控制的是"沉默多久才去敲门问对方还活着"，而不是探测间隔（那是 tcp_keepalive_intvl）或重试次数（tcp_keepalive_probes）。在半导体设备通信中，如果设备与 EAP 之间长时间无数据交互，KeepAlive 就是确保连接没有"假死"的最后一道防线。

---

📚 [Linux 性能] iostat + iotop：磁盘 I/O 的「心电图仪」

假设你在 RTD 产线上值班，突然工单延迟告警响个不停。数据库查询没变慢，CPU 也没打满，但所有写 Trace 的操作都在排队。这时候你需要两把手术刀：iostat 和 iotop。

iostat：宏观趋势图

iostat 就像医院的心电图——它告诉你磁盘的整体健康状况。最常用的命令：

  iostat -x 1 5   # 每秒输出一次扩展统计，共5次

关键指标看三个：

• await：每个 I/O 请求的平均等待时间（毫秒）。HDD 正常 5-10ms，SSD 正常 0.1-1ms。如果 await 突然飙升到 100ms+，磁盘已经在喘粗气了。

• %util：磁盘繁忙度百分比。接近 100% 说明磁盘是瓶颈。但要注意——SSD 能并行处理，%util 到 70% 就可能已经饱和了。

• r/s + w/s：每秒读写次数。如果 w/s 在 RTD Trace 写入高峰期突然从 50 跳到 5000，说明有批量数据在疯狂落盘。

打个比方：磁盘就像高速公路收费站。r/s 和 w/s 是来车的数量，await 是每辆车平均排队+缴费的时间，%util 是收费员在忙的时间比例。来车太多但只有一个收费窗口（单块 HDD），await 和 %util 双双爆炸。

iotop：揪出罪魁祸首

iostat 告诉你「磁盘很忙」，iotop 告诉你「谁在让它忙」：

  iotop -o -b -n 3   # 只显示有 I/O 的进程，批量模式输出3次

输出类似：

  PID  DISK READ  DISK WRITE  COMMAND
12456   0.00 B/s   45.23 M/s  java -jar rtd-engine.jar
23189   0.00 B/s   12.11 M/s  groovy TraceWriter.groovy

RTD 场景里，你常会看到某个 Groovy 脚本的 TraceWriter 占了一半磁盘写入——可能是某个 batch job 在全量导出历史 Trace 到 NAS。这时候你有几个选择：① 限速（在脚本里加 sleep 或批量写入控制）；② 分流（Trace 写到独立磁盘，别和应用日志抢 I/O）；③ 换 SSD（如果 HDD 的 IOPS 上限 ~150 已经不够用了）。

关联 RTD 的实际场景

在半导体 FAB 里，一台光刻机每分钟能吐出几万条 Trace 记录。如果 20 台设备同时跑，Trace 写入就是磁盘 I/O 的噩梦。正确的做法是把 Trace 目录单独挂载一块 NVMe SSD：

  mount /dev/nvme0n1 /data/trace
  iostat -x 1 | grep nvme0n1

NVMe SSD 的 IOPS 是 HDD 的上千倍（50 万 vs 150），await 从 80ms 降到 0.2ms——立竿见影。

💡 记住一句话：CPU 慢你能等，内存满你能加，磁盘 I/O 炸了——整个系统都在排队等那一块磁头转圈。

## 12:02 | 中间件

### Nacos 配置中心：让 RTD 服务「热更新」的秘密

上期答案: C — await 是 I/O 请求的平均等待时间（含队列时间和服务时间）。队列时间+服务时间，加起来就是 await。HDD 正常 5-10ms，SSD 0.1-1ms，超 100ms 就该慌了。

📚 [中间件] Nacos 配置中心：让 RTD 服务「热更新」的秘密

想象一下：你家电视没有遥控器，每次换台都要走到电视机前拧旋钮。这就是传统配置文件的工作方式——改一行配置，重启整个服务。

Nacos 就是那把遥控器。改完配置，服务自动感知，零停机更新。对 RTD 产线来说，一台设备的超时参数从 30 秒改成 60 秒，不需要停线。

为什么 RTD 需要配置中心？假设你负责 FAB 的 RTD 系统，有 20 个微服务：Recipe 管理、设备调度、Trace 采集、报警引擎……每个服务的数据库地址、日志级别、设备通信超时各不相同。用 application.yml 硬编码，光改一个 DB 密码就要跑 20 个服务各改一次、各重启一次——产线停 20 分钟。用 Nacos：在管理后台改一条配置，20 个服务同时生效。

三级隔离：Namespace（环境）→ Group（服务组）→ Data ID（具体配置）。DEV 和 PROD 用不同 namespace 物理隔绝；同一环境下不同服务用不同 group；每个配置项独立 dataId，改哪个调哪个。

代码示例（Spring Boot + Nacos）：bootstrap.yml 配置 server-addr + namespace + group，Controller 加 @RefreshScope + @Value。原理是 Nacos Client 长轮询监听变更→发 RefreshEvent→Spring Cloud 重新绑定 @Value 的值——Bean 还是那个 Bean，属性已悄悄换了。

RTD 实战：某 FAB 的 EAP 与光刻机通信偶发超时。传统做法：改配置→重启 EAP→当前 Lot 中断（损失数万）。用 Nacos：运维改 device.connect.timeout 从 30000→60000→EAP 自动感知→下一个 wafer 用新超时，当前 Lot 不受影响。

💡 记住一句话：配置不是代码，不该跟代码一起编译。Nacos 让你在飞机飞行时换引擎零件。

### Git Hooks：给 RTD 工作流装上「自动安检门」

**上期答案: C**

📚 **[Git] Git Hooks：给 RTD 工作流装上「自动安检门」**

想象你管理着 FAB 里 50 个工程师的 RTD 脚本仓库，有人把 password=123456 写进了 Groovy 脚本，有人 commit message 只写了个 "fix"，有人直接往 main 分支推代码。靠人工审查？盯不过来的。

Git Hooks 就是给你仓库装上的自动安检门——特定事件触发时（commit 前、push 前、merge 后），自动跑一段脚本来检查。

**常用 Hooks 一览：**

| Hook | 触发时机 | RTD 场景 |
|------|---------|---------|
| pre-commit | git commit 之前 | 扫描密码/Token，拒绝提交 |
| commit-msg | 写完 commit message 后 | 强制规范：如 [RTD-123] 修复Trace采集超时 |
| pre-push | git push 之前 | 跑单元测试，不通过禁止推送 |
| post-merge | merge 完成后 | 自动更新依赖、迁移数据库 |

**实操：防止密码泄露**

.git/hooks/pre-commit：
```bash
#!/bin/bash
if git diff --cached | grep -iE "password\s*=\s*['"]?\w{4,}" > /dev/null; then
    echo "❌ 发现疑似明文密码，commit 被拒绝！"
    exit 1
fi
```

**实操：强制 commit 规范**

.git/hooks/commit-msg：
```bash
#!/bin/bash
MSG=$(cat "$1")
if ! echo "$MSG" | grep -qE "^\[[A-Z]+-[0-9]+\]"; then
    echo "❌ Commit 格式错误，请使用：[RTD-123] 变更描述"
    exit 1
fi
```

**团队共享 Hooks 的坑**

Hooks 在 .git/hooks/ 下，不会随 git push 同步给其他人。正确做法：
```bash
mkdir -p .githooks
cp .git/hooks/pre-commit .githooks/
git config core.hooksPath .githooks
```

**RTD 产线实战意义**

产线变更管理是半导体软件的生命线。一个不合规的 commit 可能意味着：改了 SECS 消息解析逻辑但没留 JIRA 号，出问题根本追溯不到。Git Hooks 让你把规范从「靠自觉」变成「靠系统」——不通过安检，代码连门都进不去。

💡 **记住一句话：Git Hooks 不是给你的，是给你团队的——把规范写进脚本，比写进 Wiki 管用 100 倍。**

### Linux 进程信号与 trap 实战

📚 [Linux Shell] 进程信号与 trap 实战：让 RTD 脚本优雅退场

上期答案: B（.git/hooks/ 目录不在 Git 版本控制范围内，克隆仓库不会同步 hooks。解决办法：把 hooks 脚本放在仓库普通目录，再通过 Makefile 或 setup 脚本软链接过去。）

---

想象一下：你写了一个 RTD Trace 数据采集脚本，每 5 秒从机台捞一批数据写到磁盘。某天运维说「停一下，要维护」，直接 kill 了你的进程——结果那批还在缓冲区里的数据全丢了，下次重启还得手动补。这就是「信号处理」没做好的后果。

Linux 里，进程之间的通信除了管道、Socket，还有一套「传纸条」机制——信号 (Signal)。操作系统或用户可以通过信号告诉进程：「该歇了」「重读配置」「紧急停止」。

常见信号速记：

SIGINT (Ctrl+C)：可拦截，礼貌地请进程退出
SIGTERM (kill 默认)：可拦截，标准终止方式
SIGKILL (kill -9)：不可拦截，内核直接毙掉进程
SIGHUP：终端断开时发送，常被守护进程复用为「重载配置」
SIGUSR1/SIGUSR2：用户自定义，想怎么用怎么用

SIGKILL 是终极杀招——内核直接干掉进程，不给任何清理机会。所以优雅的程序永远用 SIGTERM 作为「请你收拾好再走」的敲门砖，SIGKILL 只兜底。

Shell 脚本里，用 trap 命令捕获信号：

```bash
cleanup() {
    sync                    # 刷磁盘缓存
    rm -f "$PID_FILE"       # 清理锁文件
    curl ...                # 通知监控下线
}
trap cleanup EXIT SIGTERM SIGINT
```

只要在脚本开头注册了 trap，无论脚本正常结束、还是被 Ctrl+C 打断、还是被 kill 发送 SIGTERM，cleanup 函数都会执行——数据不丢，状态干净。

EXIT 是 bash 的伪信号，覆盖了绝大多数退出场景（但 SIGKILL 除外，因为进程被内核直接杀，根本不执行任何代码）。

RTD 实战技巧：

① 用 SIGUSR1 做热重载——运维执行 kill -USR1 PID，脚本重读配置文件，零停机：
```bash
trap 'source /etc/rtd/config.sh; echo "配置已热重载"' SIGUSR1
```

② 用 SIGHUP 做原地重启——Nginx 就是这么干的：
```bash
trap 'exec "$0" "$@"' SIGHUP
```

③ 嵌套 trap——先清理临时文件，再发退出信号给子进程，确保整个进程树干净退出。

---

💡 记住一句话：SIGTERM 是「请你下班」，SIGKILL 是「保安架走」——永远给进程一次收拾桌子的机会。
