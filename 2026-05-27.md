---

不知道你有没有过这种体验：在产线上，晶圆在一片一片地跑，每一片该去哪台设备、跑什么配方、出什么参数，这些规则变起来比代码改版还频繁。你要是每次都用 Java 改 → 编译 → 打包 → 重启，产线早停了。

这就是 Groovy 在半导体 RTD 系统里的价值：**让业务规则可热加载**。

**闭包，就是一块你随时能打开的代码**

你可以把它想象成一把折叠伞。收起来递给同事：喏，这个工序逻辑你拿着。他到车间要用的时候，撑开就行。

```groovy
// 定义一个蚀刻工序模板
def etchStep = { wafer, params ->
    wafer.status = "ETCHED"
    log("${wafer.id} 蚀刻完成, 耗时 ${params.time}s")
}

// 用到时才执行
etchStep(wafer01, [time: 120, temp: 200])
```

闭包不只是「能传参的函数」，它有三个特殊属性：owner（定义它的对象）、delegate（方法查找目标对象）、resolveStrategy（找方法的优先级）。delegate 是 DSL 的灵魂——它让闭包里的方法调用「重定向」到另一个对象身上。

**DSL：用 Groovy 写「看起来像配置文件的东西」**

比如在 RTD 系统中，设备工程师想定义一批报警规则：

```groovy
alarmRules {
    forDevice("E-01") {
        when("温度 > 200") {
            severity "CRITICAL"
            notify "shift-leader@factory.com"
            action "HOLD_EQUIPMENT"
        }
        when("压力 < 0.1") {
            severity "WARNING"
            autoRecoverAfter 5.minutes  // Groovy 语法糖
        }
    }
}
```

这看起来是人写的配置，但它其实是合法的 Groovy 代码，背后靠的就是把闭包的 delegate 指向一个 DSL 上下文对象：

```groovy
def alarmRules(Closure block) {
    block.delegate = new AlarmDSLContext()
    block.resolveStrategy = Closure.DELEGATE_FIRST
    block.call()
}
```

当解析到 `when(...) { ... }` 时，Groovy 在闭包自己的类里找不到 `when` 方法 → 顺着 delegate 找 → 在 AlarmDSLContext 里找到并执行。

**在 RTD 场景里到底怎么用？**

调度引擎（哪台设备空闲了派哪个批次），MES 工序流转规则，设备 PM 保养触发条件，SPC 统计判定规则——这些都应该写成 .groovy 文件放在固定目录。系统通过 GroovyShell 或 GroovyScriptEngine 热加载，改动后无需重启：

```
运维改 .groovy → 文件时间戳变了 → 引擎自动重载 → 新规则生效
```

这就是半导体柔性制造的核心基础设施之一。

💡 记住一句话：**闭包是 Groovy 的肌肉，delegate 是骨架，DSL 是穿上西装后的样子——看着像配置，跑起来是代码。**

[网络] TCP KeepAlive 与心跳：半导体设备通信的「呼吸检测」

在半导体 Fab 里，RTD 系统通过 TCP 长连接跟几百台设备通信——EAP 发指令、收 SECS 消息、采集 Trace 数据。问题来了：网络断了你得立刻知道。不然设备挂了你还以为连着，Lot 卡在机台里没人管，这就是生产事故。

TCP 自带 KeepAlive 机制。就像你打电话时沉默太久，对方问「还在吗？」——TCP 空闲后发探测包，没回应就重试，再没回应就判死亡。

三个关键参数（Linux）：
net.ipv4.tcp_keepalive_time = 7200   # 空闲多久开始探测，默认2小时
net.ipv4.tcp_keepalive_intvl = 75    # 探测间隔（秒）
net.ipv4.tcp_keepalive_probes = 9    # 探测几次就放弃

默认要 7200+75*9=7875 秒（>2小时）才发现断连——在产线上完全不可接受！

所以 RTD 系统必须叠加应用层心跳：客户端每 10 秒发 PING，服务端回复 PONG。3 次没回（30秒）就触发重连。Netty 实战一行搞定：

ch.pipeline().addLast(new IdleStateHandler(0, 0, 10, TimeUnit.SECONDS));
// 10 秒没收到数据 -> userEventTriggered 发心跳帧

同时开启 Socket 级 KeepAlive 兜底：
bootstrap.option(ChannelOption.SO_KEEPALIVE, true);

TCP KeepAlive 是兜底安全网，应用层心跳才是生命线。在 RTD 场景，宁可误判重连（多几条日志），也不能漏判（卡 Lot 就是生产事故）。

记住一句话：TCP KeepAlive 默认 2 小时才发现断连，RTD 必须叠加应用层心跳才能守住产线。

## 11:03 | Linux 性能

### iostat + iotop：磁盘 I/O 的「心电图仪」— RTD Trace 落盘性能诊断

上期答案: B

tcp_keepalive_time 的默认值是 7200 秒（2 小时），含义是 TCP 连接在空闲多久后开始发送第一个 KeepAlive 探测包。这个参数控制的是"沉默多久才去敲门问对方还活着"，而不是探测间隔（那是 tcp_keepalive_intvl）或重试次数（tcp_keepalive_probes）。在半导体设备通信中，如果设备与 EAP 之间长时间无数据交互，KeepAlive 就是确保连接没有"假死"的最后一道防线。

📚 [Linux 性能] iostat + iotop：磁盘 I/O 的「心电图仪」

假设你在 RTD 产线上值班，突然工单延迟告警响个不停。数据库查询没变慢，CPU 也没打满，但所有写 Trace 的操作都在排队。这时候你需要两把手术刀：iostat 和 iotop。

iostat：宏观趋势图

iostat 就像医院的心电图——它告诉你磁盘的整体健康状况。最常用的命令：

  iostat -x 1 5   # 每秒输出一次扩展统计，共5次

关键指标看三个：

• await：每个 I/O 请求的平均等待时间（毫秒）。HDD 正常 5-10ms，SSD 正常 0.1-1ms。如果 await 突然飙升到 100ms+，磁盘已经在喘粗气了。

• %util：磁盘繁忙度百分比。接近 100% 说明磁盘是瓶颈。但要注意——SSD 能并行处理，%util 到 70% 就可能已经饱和了。

• r/s + w/s：每秒读写次数。如果 w/s 在 RTD Trace 写入高峰期突然从 50 跳到 5000，说明有批量数据在疯狂落盘。

打个比方：磁盘就像高速公路收费站。r/s 和 w/s 是来车的数量，await 是每辆车平均排队+缴费的时间，%util 是收费员在忙的时间比例。来车太多但只有一个收费窗口（单块 HDD），await 和 %util 双双爆炸。

iotop：揪出罪魁祸首

iostat 告诉你「磁盘很忙」，iotop 告诉你「谁在让它忙」：

  iotop -o -b -n 3   # 只显示有 I/O 的进程，批量模式输出3次

输出类似：

  PID  DISK READ  DISK WRITE  COMMAND
12456   0.00 B/s   45.23 M/s  java -jar rtd-engine.jar
23189   0.00 B/s   12.11 M/s  groovy TraceWriter.groovy

RTD 场景里，你常会看到某个 Groovy 脚本的 TraceWriter 占了一半磁盘写入——可能是某个 batch job 在全量导出历史 Trace 到 NAS。这时候你有几个选择：① 限速（在脚本里加 sleep 或批量写入控制）；② 分流（Trace 写到独立磁盘，别和应用日志抢 I/O）；③ 换 SSD（如果 HDD 的 IOPS 上限 ~150 已经不够用了）。

关联 RTD 的实际场景

在半导体 FAB 里，一台光刻机每分钟能吐出几万条 Trace 记录。如果 20 台设备同时跑，Trace 写入就是磁盘 I/O 的噩梦。正确的做法是把 Trace 目录单独挂载一块 NVMe SSD：

  mount /dev/nvme0n1 /data/trace
  iostat -x 1 | grep nvme0n1

NVMe SSD 的 IOPS 是 HDD 的上千倍（50 万 vs 150），await 从 80ms 降到 0.2ms——立竿见影。

💡 记住一句话：CPU 慢你能等，内存满你能加，磁盘 I/O 炸了——整个系统都在排队等那一块磁头转圈。
